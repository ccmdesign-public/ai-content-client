## Reliability, Failures & Recovery

### 1\. What are all the ways my system can fail under real traffic?

***Resource Exhaustion***

-   *DB Overload*: High traffic and complex queries or too many concurrent database connections, query timeouts, locked tables or total db crash
-   *CPU Memory Saturation*: Servers run out of memory, cpu max out, stalls the application
-   *Network bottleneck*: Bandwidth limitations , load balancers , firewall, ISP, traffic flow, high latency, dropped packets
-   *Disk I/O bottlenecks*: Excessive logging, writing to disk, high activity leads to I/O saturation, slowing all reads and write operations

***Software application failures***

-   *Cascading failures*: minor failure in one service slow cache consumes all available threads in parent, causing dynamo effect bring down the entire application
-   *Concurrency issues*: under high load, threads may conflict over shared resources, data corruption, deadlocks, unpredictable behavior
-   *Poison pills*: specifically malformed request causes a service to crash, repeated failure if the request is repeated
-   *In-efficient code or algorithms*: exponential slow with 10k users, leading to O(n^2) operations problems

***Infra and dependency failures***

-   *Third party service failure*: external dependencies blocking users
-   *Cold start / autoscaling features*: new server takes too long to start, auto scaling lags, traffic surge, downtime
-   *DNS Failures*: Service unreachable because of DNS attacks
-   *Hardware Failures*: Server, RAM, disk or network interface cards fail due to intense pressure and head generated by high loads

***Human and process Failures***

-   *Misconfiguration*: Improper load balancers, rate limits, data base pool size exposed only by traffic spikes
-   *Failed deployments*: New code release contains bugs, only surface under load, stressful rollback
-   *Cache miss storms*: Cache expires simultaneously for many items, massive simultaneous surge in db queries

***Malicious or unexpected traffic***

-   *DDOS Attacks*: Intentional floods of requests to exhaust resources
-   *Reddit Hug*: Surge of legitimate traffic cause sudden spike in popularity that exceeds capacity
-   *Bot Traffic*: Not human traffic, scrapers, malicious bots over load specific endpoints, login before traffic hits

***Signs***

-   *503 Service Unavailable*: System too overwhelmed to process requests
-   *High Latency*: Pages takes too long to load, causing delay
-   *504 Gateway Timeout*: Backend took too long to answer
-   *Data in-consistency*: Users see old data, or transactions are partially completed

***Prevention strategies***

-   *Load testing*: Test the system with expected peak traffic before it happens
-   *Rate limiting*: Protect limits with number of request per user
-   *Caching*: Use CDN and in memory cache to reduce db load
-   *Graceful degradation*: Disabling no essential features like recommendations to keep core functionality alive

### 2\. How do I design for graceful degradation instead of hard failure?

-   *Identify Critical Functions*: Determine the essential services that should remain functional, and allow non essential features to fail or be disabled
-   *Implement Circuit Breaker and Retries*: Prevent overloading by breaking connections and automatically using cached or placeholder data to stay responsive
-   *Use Feature Flags*: Quickly deactivate, disable, or replace failing features requiring redeployment
-   *Implement Load Shedding*: During high traffic, sacrifice less critical functionality, e.g recommendation engines, prioritize critical features like checkout
-   *Design for failure*: Assume dependencies will fail, design alternative, simplified paths for users rather than displaying error page
-   *Clear communication*: Inform users that specific features are temporarily unavailable, rather than experience hard failure and maintain trust

*Examples*

-   *Frontend*: Use html/css with alt text which degrades gracefully
-   *Backend*: Search service failing over to a cached, slightly outdated version instead of returning HTTP 500 error
-   *System*: E-commerce site disabling recommendation to keep the payment system running

### 3\. How do I implement retries without causing storms and loops?

*Backoff + Jitter*

-   *Exponential Backoff*: Instead of retrying use escalating delay increase the wait time exponentially after failure (1s, 2s, 4s,8s, 16s)
-   *Jitter*: Add a random amount of time to the backoff, breaks the synchronization, preventing thousands of clients from retrying at the exact same time

***delay = min(cap \* 2 \*\* attempt)
sleep\_time = delay \* (0.5 + random \* 0.5) # jitter***

*Preventing Infinite loops*

-   *Set finite limits*: limit attempts, 3–5 attempts
-   *Set maximum delay*: Cap the maximum time between retries (don’t wait more than 30 seconds)
-   *Use timeouts*: Set a timeout api request itself to ensure you aren’t waiting too long for a response

*Making operations Idempotent*

-   *Idempotency keys*: POST requests which fails due to network timeout, should not be allowed to retry again. Generate unique UUID for each distinct action, send this ID in the request header, *Idempontency-key: <UUID>*
-   *Server side tracking*: the server must store processed keys (e.g Redis), and return the same response for the same key rather than re running the operation

*Advanced Protection Measures*

-   *Circuit Breaker*: If service fails consistently, open circuit breaker to stop all outgoing requests allowing time for the service to recover. only try after cool down period
-   *Retry Budgets*: limit the number of retries allowed relative to the total volume of requests (retries should not exceed 10% of total traffic)
-   *Respect Retry-After*: If the server returns 429 (Too many requests) or 503 (Service unavailable) with a ***Retry-After*** header, honor it and wait that specific amount of time

### 4\. How do I design circuit breakers and bulkheads correctly?

***Circuit Breaker:*** Safety mechanisms that detect faults, overloads, service failures and interrupt the flow to prevent damage

-   *Three states*: Open (Failure Detected), Half open (Test for Recovery), Close (Normal operation)
-   *Thresholds*: Set failure rate thresholds based on criticality, along with the sliding window size for calculation
-   *Recovery timeout*: Define a reset timeout that allows the system to recover before retesting

***Bulkhead:*** Partitions that create isolated, compartment preventing single failure from flooding the entire system

-   *Resource Isolation*: Pool Resources (separate thread pool, database connections) for different services so one slow service doesn’t exhaust capacity for others
-   *Singleton pattern*: Use single instances of bulkheads per service to ensure limits are applied consistently.

### 5\. How do I run in multiple regions and fail over cleanly?

***Multi Region Failover***

-   *Traffic Management*: use DNS based failover, Route 53 with low TTL (20–60 seconds) and health checks to automatically shift traffic from a failed region
-   *Data consistency*: Implement cross region replication from databases to ensure data is available in the standby region
-   *Infrastructure as code (IAC)*: Use tools like transform or AWS cloud formation to maintain identical, consistent environments in both regions
-   *Application Region*: Ensure applications are stateless and can immediately start serving traffic in the new region
-   *Automated Testing*: Regularly perform chaos engineering and failover drills to ensure standby region can handle full load and the automation works as expected

*Failover Procedure*

-   *Detect* : Health checks to identify primary region failure
-   *Promote*: Standby to Primary
-   *Redirect Traffic*: Update DBS records or load balancer weights to route user traffic to the healthy region

### 6\. How do I design for data center outage scenarios?

***Redundant Infrastructure***

-   *Power*: implement uninterrupted power supplies (UPS), backup generators and dual utility feeds from different substation for continuous operation during failure
-   *Cooling*: Deploy Redundant Cooling Units -> N+1 cap to prevent failure due to overheating
-   *Network*: Utilize diverse fiber paths entering the building to prevent service loss from cable cuts
-   *Site Mirroring*: Maintain two, geographically separated data center with real time data replication to allow seamless failovers
-   *Cloud/Hybrid Integration*: Utilize cloud load balancing to redirect traffic to alternative zones or region if a primary site fails

***Disaster Recovery Planning***

-   *Define Objectives*: Establish strict recovery time objectives (RTO — how fast to restore) and Recovery Point Objectives (how much data loss is acceptable)
-   *Automated Run-books*: Create automated scripts for failover procedures to minimize human error and downtime
-   *Testing & Drills*: Conduct regular, simulated or live fail over drills to ensure the team and systems are prepared

***Preventive Measures***

-   *Risk Assessment*: Conduct comprehensive audits to identify single point of failure
-   *Monitoring & Maintenance*: Implement 24/7 monitoring and regular , scheduled maintenance on all critical components

### 7\. How do I do schema changes without downtime?

-   *Phase 1 (Expand)*: Add new columns, tables or constraints without removing old ones. the db supports both old and new code
-   *Phase 2 (Migrate / Deploy)*: Update the application code to use the new schema while writing to both old and new columns. Run background script to migrate existing data
-   *Phase 3 (Contract)*: Once all instances are running new code, remove old schema components e.g old columns or tables
-   *Non Blocking Operations*: Avoid commands that locks table for long periods (like ALTER table). use online schema change tools like ***pt-online-schema-change*** for mysql or ***pg-online-schema-change*** for postgres
-   *Backfill data*: When renaming or changing columns, add new column, sync data via triggers or application code then delete old columns
-   *Safe Not Null constraints*: In postgres, add a constraint using not valid, then validate it in a second step to avoid locking the table
-   *Feature flags*: Use to toggle between old and new db code allowing rollbacks

### 8\. How do I design a rollback strategy for bad releases?

-   *Pre deployment preparation*: Implement versioning for all artifacts, database schemas and configurations
-   *Automated monitoring*: Set up alerts for error rates, latency or failed health checks to trigger automatic rollbacks

***Deployment techniques***

-   *Feature flags*: Wrap new features in flags to disable them instantly without a full redeployment
-   *Blue green deployment*: Maintain two identical environments, switch traffic to the new version ,with the ability to instantly switch back th the old (blue)
-   *Canary deployment*: Release to a small percentage of users first to limit the blast radius of a failure
-   *Backward Compatibility*: Ensure database changes are backward compatible such as new adding new columns rather than renaming old ones, allowing the old code to still work if a rollback is necessary
-   *Run-book and testings*: create clear, step by step documentation for the rollback procedure and practices these steps regularly

*Rollback*: when the error is critical, widespread or causing significant data issues reverts to last known stable state
*Roll-Forward*: if the issue is minor, quick fix involves deploying new fix on top of broken release

Best Practice is to Automate, test and validate in the staging environment, keep the changes smaller and frequent deployments reduce the risk of rollback

### 9\. How do I make “rebuild from scratch” possible if everything is lost?

***Immediate Stabilization***

-   *Stop all overwrites*: Immediately power down machines or remove drives where data was lost to prevent further data being overwritten
-   *Search for hidden backups*: Check personal drives, developer laptops, old mails, staging servers
-   *Use forensic data recovery*: use specialized data recovery software to scan for deleted files on hard drives or SSD’s

***Forensic Reconstruction (If binary Exists)***

-   *Decompilation*: For languages like java or C# use decompilers to turn machine code back into high level source code
-   *Disassembly*: For native code , use disassembly tools to convert binary back to assembly language
-   *Extraction*: For .Net applications, you can copy forms and controls from the running application to a new project, as metadata often remains

***Rebuilding from scratch (If Binary does not exists)***

-   *Gap Analysis*: Document the intended state vs what is actually needed identifying the gaps to create a roadmap
-   *Document functionality*: Interview users, stakeholders, and developers to map out user flow and business logic
-   *Implement “Infrastructure as code”*: Define new environment using tools that allows automated rebuilding ensuring this type of loss doesnt happen again.

***Strategy for Revival***

-   *Incremental Rebuilding*: Do not rebuild everything at once. use small advances only with two week release cycles
-   *Re-verify data logic*: If databases were saved but code was not , use the remaining data to infer schema structures
-   *Modernize*: Treat the loss as an opportunity to upgrade the technology stack and improve system architecture.

***Future Prevention***

-   *Use version control*: Ensure all code is store in off site repositories
-   *Implement Automated Backups*: use hybrid, cloud based and local backups and test them regularly.
-   *Knowledge redundancy*: Avoid single point of failure by ensuring documentation is shared among the team

### 10\. How do I define SLOs and design around them?

***What is SLO — Service Level Objectives?***

-   *Identify Critical User Journey (CUJ’s)*: Focus on the core tasks user performs, login, check out, search for products
-   *Select Service Level Indicators (SLI’s)*: Chose metrics that measures the success of these journeys from the users perspective
-   *Availability*: service up, successful HTTP requests / total requests
-   *Latency*: Is the service fast enough (99th percentile of request < 300ms)
-   *Quality / Correctness*: Is the data accurate
-   *Define SLI Target*: Set realistic attainable goal (99.9% of requests succeed over a 30 day window) Avoid five nines unless necessary
-   *Establish Error budgets*: This is the inverse of your SLO. if 99.9% SLO — then 0.1% allowed failure. The error budget is the amount of unreliability you can tolerate allowing innovation and conversely signaling when to halt new releases to focus stability

***How to design around SLO?***

-   *Build resilience*: Deploy across multiple availability zones and implement load balancing
-   *Graceful degradation*: Design systems that provides reduced functionality rather than failing completely during partial outages (e.g caching, fallback content)
-   *Circuit Breakers*: Prevent cascading failures by protecting downstream services when a dependency is struggling
-   *Implement SLI Based Alerting*: Alert on the rate at which you are consuming the error budget rather than just raw error rates. Use multi window, multi burn rate alerts to distinguish between quick spikes and long term, slow burning issues
-   *Use error budgets to guide development*
    ***Budget Exists***: proceed with feature development and deployments
    ***Budget depleted***: Halt new releases and focus engineering effort on reliability improvements, performance tuning , bug fixes
    ***Monitor at the edge***: Measure SLI’s as close to the user as possible (API gateway or load balancer) to ensure measurement reflects actual user experience

*Best Practice*

-   *Start Simple*: Start with 2–3 key metrics per service, such as latency and availability
-   *Iterate and refile*: Review SLO performance monthly. Adjust thresholds if they are too noisy or too loose
-   *Document everything*: Maintain clear documentation on how SLI’s are calculated including what exception are excluded
