# feat: Newsletter archive as homepage + Resend signup

## Overview

Restructure the Nuxt client so the homepage (`/`) becomes the newsletter landing page: a hero section with a Resend-powered email signup form, followed by a chronological archive of past newsletter issues (newest first). Each issue gets a detail page at `/issues/[id]` rendering the full newsletter content (editorial intro, featured picks with commentary, quick links, and closing text).

This is the CTA destination for all LinkedIn posts and the primary entry point for the site. It supersedes the previous scope where the archive lived at `/newsletter/`.

**Related issues:**
- AIC-24 (archive portion -- superseded by this plan)
- AIC-30 (summaries relocation -- already merged, summaries now at `/summaries/`)
- AIC-28 (this issue)

**Related plans:**
- `docs/plans/2026-03-19-001-feat-newsletter-3x-cadence-linkedin-teaser-plan.md` -- covers the scraper-side newsletter pipeline; this plan covers the client-side rendering and signup

## Problem Statement / Motivation

**Current State:**
- The homepage (`/`) is a stub page with a 302 redirect to `/summaries/` (see `src/pages/index.vue` and `src/nuxt.config.ts` routeRules)
- No newsletter archive exists in the client -- newsletter drafts are generated by `ai-content-scraper` but not rendered anywhere
- No email signup mechanism exists -- the Resend API is used for sending newsletters but there is no audience-building signup form
- LinkedIn posts have no natural CTA destination for newsletter signups

**Desired State:**
- Homepage is a newsletter landing page with hero + signup + archive
- Each newsletter issue has a detail page with full content
- Visitors from LinkedIn posts can sign up and browse past issues
- Navigation reflects the newsletter-first focus

**Why This Matters:**
- Every LinkedIn post needs a CTA link; the homepage is the natural destination
- Newsletter signup is the primary audience-building mechanism ($0 acquisition)
- Archive provides social proof and value demonstration to potential subscribers

## Proposed Solution

### High-Level Approach

Four implementation phases, each independently deployable:

1. **Phase 1:** Newsletter content collection -- define the Nuxt Content collection and convert scraper draft JSONs to content files
2. **Phase 2:** Homepage redesign -- hero section + signup form + issue archive list
3. **Phase 3:** Issue detail pages -- `/issues/[id]` route with full newsletter content rendering
4. **Phase 4:** Navigation update + cleanup -- update nav, remove homepage redirect, prerender routes

### Data Source: NewsletterDraft JSON

Content is sourced from the scraper's `NewsletterDraft` JSON files (generated in `ai-content-scraper`). The schema includes:

```typescript
// From ai-content-scraper -- the source schema we must accommodate
interface NewsletterDraft {
  subjectLine: string
  editorialIntro: string
  featuredPicks: Array<{
    item: ContentItem    // title, url, source, summary
    commentary: string   // editor's commentary on this pick
  }>
  quickLinks: Array<{
    item: ContentItem
    oneLiner: string     // brief one-liner description
  }>
  closing: string
  generatedAt: string    // ISO datetime
}
```

These JSONs need to be converted to Markdown files (with YAML frontmatter) for Nuxt Content ingestion, or loaded as `type: 'data'` JSON collections.

### Implementation Phases

#### Phase 1: Newsletter Content Collection

**Goal:** Define the Nuxt Content collection and create the content directory structure so newsletter data is queryable.

**Tasks:**

- [ ] Create `src/content/newsletters/` directory
- [ ] Add `newsletters` collection to `content.config.ts`:
  ```typescript
  // content.config.ts -- add to collections object
  newsletters: defineCollection({
    type: 'page',   // 'page' gives us body rendering; 'data' if we store pure JSON
    source: {
      include: 'newsletters/*.md',
      cwd: contentDir
    },
    schema: z.object({
      subjectLine: z.string(),
      editorialIntro: z.string(),
      featuredPicks: z.array(z.object({
        title: z.string(),
        url: z.string().url(),
        source: z.string(),
        summary: z.string(),
        commentary: z.string(),
      })),
      quickLinks: z.array(z.object({
        title: z.string(),
        url: z.string().url(),
        source: z.string(),
        oneLiner: z.string(),
      })),
      closing: z.string(),
      publishedAt: z.string(),   // ISO date string (derived from generatedAt)
    })
  })
  ```
- [ ] Create `scripts/sync-newsletters.ts` -- a Node script that reads `NewsletterDraft` JSON files from a source directory and converts them to Markdown files with YAML frontmatter in `src/content/newsletters/`
  - Input: directory of JSON files (e.g., `../ai-content-scraper/data/newsletter-drafts/`)
  - Output: one `.md` file per newsletter in `src/content/newsletters/YYYY-MM-DD.md`
  - Frontmatter: all structured data (subjectLine, featuredPicks, quickLinks, closing, publishedAt)
  - Body: the `editorialIntro` as Markdown body content (enables `ContentRenderer` for rich text)
  - Idempotent: skip files that already exist (by date)
- [ ] Create 1-2 sample newsletter content files for development/testing
- [ ] Run `npm run postinstall` to verify Nuxt Content picks up the new collection
- [ ] Add unit test: validate that the sample content file passes the Zod schema

**Decision: `type: 'page'` vs `type: 'data'`**

Use `type: 'page'` with Markdown files. This lets us use `ContentRenderer` for the editorial intro (which may contain rich text formatting), while structured data lives in frontmatter. The alternative (`type: 'data'` with JSON files) would be simpler but loses rich text rendering for the intro and closing.

#### Phase 4: Resend Signup API + Navigation + Cleanup

**Goal:** Wire up the Resend audience API, update navigation, and finalize prerender configuration.

**Tasks:**

- [ ] Add Resend runtime config to `src/nuxt.config.ts`:
  ```typescript
  runtimeConfig: {
    resendApiKey: process.env.RESEND_API_KEY || '',
    resendAudienceId: process.env.RESEND_AUDIENCE_ID || '',
    // ... existing config
  }
  ```
- [ ] Create `src/server/api/subscribe.post.ts`:
  ```typescript
  export default defineEventHandler(async (event) => {
    const body = await readBody<{ email?: string }>(event)
    const email = (body?.email || '').trim()

    // Validate email format
    if (!email || !/.+@.+\..+/.test(email)) {
      throw createError({ statusCode: 422, statusMessage: 'Invalid email address' })
    }

    const config = useRuntimeConfig()
    if (!config.resendApiKey || !config.resendAudienceId) {
      throw createError({ statusCode: 500, statusMessage: 'Email service not configured' })
    }

    // Add contact to Resend audience
    const response = await $fetch(
      `https://api.resend.com/audiences/${config.resendAudienceId}/contacts`,
      {
        method: 'POST',
        headers: {
          Authorization: `Bearer ${config.resendApiKey}`,
          'Content-Type': 'application/json',
        },
        body: { email, unsubscribed: false },
      }
    )

    return { ok: true }
  })
  ```
  - Uses Resend's REST API directly (no SDK dependency needed -- it is a single POST call)
  - Pattern follows existing `src/server/api/contact.post.ts`
  - Error handling: catch Resend API errors and return user-friendly messages
  - Rate limiting consideration: Resend allows 2 req/sec; for a signup form this is not a bottleneck

## System-Wide Impact

### Interaction Graph

```
User visits / (homepage)
  -> src/pages/index.vue
    -> queryCollection('newsletters') -- fetches all issues
    -> renders NewsletterSignupForm (client-side)
    -> renders IssueCard list

User submits signup form
  -> NewsletterSignupForm.vue
    -> $fetch('/api/subscribe', POST)
      -> src/server/api/subscribe.post.ts
        -> Resend API: POST /audiences/{id}/contacts

User clicks issue
  -> src/pages/issues/[id].vue
    -> queryCollection('newsletters').path(...)
    -> renders full newsletter content

Build/deploy
  -> nuxt generate
    -> prerender / (homepage)
    -> crawlLinks discovers /issues/* pages
    -> sitemap.xml includes issue URLs
```

### Error Propagation

- **Resend API failure** in subscribe endpoint: caught in the event handler, returns 500 with user-friendly message. Frontend shows error state. No retry logic needed (user can resubmit).
- **Missing Resend config**: server returns 500 "Email service not configured". This is a deployment issue, not a user error. The signup form remains visible but non-functional -- consider showing a fallback message if the form is permanently broken.
- **Newsletter collection empty**: homepage gracefully shows "First issue coming soon!" empty state. No errors thrown.
- **Invalid newsletter content file**: Zod schema validation will fail at build time during `nuxt generate`, surfacing the issue early.

### State Lifecycle Risks

- **Duplicate signups**: Resend's audience API handles duplicate contacts gracefully (upserts rather than errors). No dedup logic needed on our side.
- **Static site + dynamic signup**: The homepage is prerendered (static), but the signup form makes a POST to a server route. In `static` preset, Nitro server routes are not available at runtime. **This is a critical consideration** -- see Open Questions.

| Risk | Impact | Mitigation |

## Open Questions

1. **Static site + server routes**: The current Nitro preset is `static`. Server API routes (like `subscribe.post.ts`) do not exist at runtime in a static deployment. Options:
   - **A)** Switch Nitro preset to `netlify` (or the appropriate hosting preset) so server routes become serverless functions
   - **B)** Keep `static` and use Netlify Functions separately for the subscribe endpoint
   - **C)** Use a client-side-only approach (call Resend API directly from the browser -- but this exposes the API key)
   - **Recommendation:** Option A -- switch to `netlify` preset. This is the simplest path and Netlify already hosts the site. The preset handles both prerendering and serverless functions.

2. **Newsletter content sync workflow**: How will newsletter content files get into this repo?
   - **A)** Manual: copy JSON files and run `scripts/sync-newsletters.ts` locally, then commit
   - **B)** CI: scraper repo triggers a webhook/CI job that syncs content to this repo
   - **C)** Build-time: fetch from scraper's data directory at build time
   - **Recommendation:** Start with A (manual), automate with B later. The sync script makes A low-friction.

3. **Newsletter naming / branding**: The hero section needs a newsletter name and tagline. The scraper plan mentions a brand name decision is pending (Phase 3 of the cadence plan). Use a placeholder ("AI Content Digest") and update when finalized.

4. **Issue URL scheme**: `/issues/[id]` where `[id]` is the date slug (e.g., `2026-03-19`). Confirm this is the desired format vs. sequential numbering (`/issues/42`) or subject-based slugs.
