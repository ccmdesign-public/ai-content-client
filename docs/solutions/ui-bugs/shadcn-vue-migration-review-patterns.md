# shadcn-vue Migration Code Review Patterns

## Problem

After migrating content components to shadcn-vue primitives (PR #13), a code review surfaced 9 issues across 5 distinct pattern categories. These are recurring pitfalls when adopting shadcn-vue in a Nuxt SSR project and are worth documenting as a migration checklist.

## Root Cause Analysis

The issues fell into 5 categories, each with a distinct root cause:

### 1. shadcn-vue CLI silently removes `framework` config

**What happened:** Running `pnpm dlx shadcn-vue@latest add <component>` overwrote `components.json` and dropped the `"framework": "nuxt"` field. Future CLI invocations would then generate components with manual Vue imports instead of Nuxt auto-imports.

**Why:** The shadcn-vue CLI regenerates `components.json` from its own template, which may not include fields it does not recognize or that were added manually.

**Fix:** Re-add `"framework": "nuxt"` to `components.json`. Check this field after every CLI-based component install.

### 2. Tailwind `group-hover` requires explicit `group` class on ancestor

**What happened:** `IssueCard.vue` used `group-hover:translate-x-0.5` and `group-hover:text-primary` on an arrow icon, but the parent `NuxtLink` lacked the `group` class. The hover animation never triggered.

**Why:** Tailwind's `group-hover` variant only activates when an ancestor with class `group` is hovered. Without it, the utility is silently dead CSS.

**Fix:** Add `group` to the parent element's class list. This is a single-line change.

**Prevention:** When reviewing any `group-hover:*` or `group-focus:*` usage, always verify the nearest `group`-classed ancestor exists.

### 3. SSR-unsafe `document.cookie` in shadcn-generated components

**What happened:** `SidebarProvider.vue` (generated by shadcn-vue) writes to `document.cookie` in its `setOpen()` function without an SSR guard. While current code paths only trigger this client-side, future programmatic sidebar state changes during SSR would crash with `ReferenceError: document is not defined`.

**Why:** shadcn-vue generates components targeting browser environments. Nuxt SSR requires explicit guards around browser-only APIs.

**Fix:** Wrap the `document.cookie` write in `if (import.meta.client) { ... }`. This is Nuxt's idiomatic SSR guard.

**Prevention:** After installing any shadcn-vue component, grep for `document.`, `window.`, and `navigator.` and add `import.meta.client` guards where needed.

### 4. Dead page meta accumulates after layout refactors

**What happened:** The `default.vue` layout was refactored to no longer read `route.meta.hero`, but 11 pages still declared `hero: false` or `hero: { ... }` in `definePageMeta()`. This dead configuration confused developers about what `hero` does.

**Why:** Layout changes that remove a feature (hero rendering) do not automatically clean up the consumers (pages setting the meta). This is a common oversight in framework-convention-based architectures where the contract is implicit.

**Fix:** Remove all `hero` properties from `definePageMeta()` calls across all pages.

**Prevention:** When removing a feature from a layout, grep for the corresponding `route.meta.*` key across all pages and clean up references in the same PR.

### 5. Legacy Vue 2 dependencies linger after partial cleanup

**What happened:** `vue2-perfect-scrollbar` remained in `package.json` even though no code imported it. The PR had already removed `vue-carousel` for the same reason but missed this one.

**Why:** Dependency cleanup is manual and easy to miss when multiple legacy packages exist. `pnpm` does not warn about unused dependencies by default.

**Fix:** Remove the dependency from `package.json`.

**Prevention:** After any migration that removes Vue 2 components, run `grep -r "vue2-" package.json` and check for zero-import dependencies. Consider using `depcheck` or `knip` for automated unused dependency detection.

## Additional Issues Resolved

- **Duplicate `components/ui` registration** in `nuxt.config.ts` (scanned twice, registered thrice counting shadcn-nuxt module)
- **Unused `Separator` import** in `DateGroupedFeed.vue` (planned but never used in template)
- **Duplicated 250-char Tailwind class string** in `CategoryFilterBar.vue` (extracted to `const chipClass`)
- **Zod v4-to-v3 downgrade** verified as safe: all usage (`z.object()`, `z.string()`, `z.infer`, `z.ZodError`) is standard v3 API. The downgrade was caused by `@vee-validate/zod` peer dependency requiring `zod ^3.24.0`.

## Prevention Checklist for Future shadcn-vue Migrations

1. After `shadcn-vue add`: verify `components.json` still has `"framework": "nuxt"`
2. After installing any UI component: grep for `document.`, `window.`, `navigator.` and add SSR guards
3. After layout refactors: grep for removed `route.meta.*` keys across all pages
4. After removing legacy components: check for orphaned Vue 2 dependencies
5. Review all `group-hover:*` / `group-focus:*` usage to verify `group` ancestor exists
6. Check for duplicate entries in `nuxt.config.ts` `components` array
7. Check for unused imports that were planned but never wired to template
